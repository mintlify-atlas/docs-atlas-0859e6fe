## Contribution Workflow

<Steps>
  <Step title="Fork the Repository">
    Start by forking the Discord OTP Bot repository to your GitHub account.
    
    ```bash
    # Visit the repository
    https://github.com/2577mamneet/discord-otp-bot
    
    # Click the "Fork" button in the top right
    ```
    
    Then clone your fork locally:
    
    ```bash
    git clone https://github.com/YOUR_USERNAME/discord-otp-bot.git
    cd discord-otp-bot
    ```
  </Step>
  
  <Step title="Create a Feature Branch">
    Create a new branch for your changes. Use a descriptive name that reflects what you're working on.
    
    ```bash
    git checkout -b feature/YourFeature
    ```
    
    Branch naming conventions:
    - `feature/` - for new features (e.g., `feature/add-yahoo-support`)
    - `fix/` - for bug fixes (e.g., `fix/imap-timeout-error`)
    - `docs/` - for documentation changes (e.g., `docs/update-readme`)
    - `refactor/` - for code refactoring (e.g., `refactor/otp-extraction`)
  </Step>
  
  <Step title="Make Your Changes">
    Implement your feature or fix. Follow the existing code style and conventions.
    
    <Accordion title="Code Style Guidelines">
      - Use Python 3.8+ syntax
      - Follow PEP 8 style guidelines
      - Add docstrings to functions and classes
      - Include comments for complex logic
      - Use meaningful variable names
      - Keep functions focused on a single responsibility
    </Accordion>
    
    <Accordion title="Testing Your Changes">
      1. Set up your development environment:
      
      ```bash
      # Install dependencies
      pip install -r requirements.txt
      
      # Configure your .env file
      cp .env.example .env
      # Edit .env with your credentials
      ```
      
      2. Test the bot thoroughly:
      
      ```bash
      # Run the bot
      python otp_bot.py
      
      # Test all slash commands
      # - /otp
      # - /quick_otp
      # - /ping
      ```
      
      3. Verify your changes work with different email providers (Gmail, Outlook, Yahoo, etc.)
    </Accordion>
  </Step>
  
  <Step title="Commit Your Changes">
    Write clear, descriptive commit messages that explain what and why.
    
    ```bash
    git add .
    git commit -m "Add support for Yahoo Mail IMAP configuration"
    ```
    
    Good commit message examples:
    - ✅ "Fix OTP extraction regex for Instacart emails"
    - ✅ "Add support for iCloud IMAP settings"
    - ✅ "Improve error handling for IMAP connection failures"
    - ❌ "Update code"
    - ❌ "Fix bug"
    - ❌ "Changes"
  </Step>
  
  <Step title="Push to Your Fork">
    Push your feature branch to your GitHub fork.
    
    ```bash
    git push origin feature/YourFeature
    ```
    
    If this is your first push of this branch, Git will provide the command to set up tracking:
    
    ```bash
    git push -u origin feature/YourFeature
    ```
  </Step>
  
  <Step title="Submit a Pull Request">
    Go to the [original repository](https://github.com/2577mamneet/discord-otp-bot) on GitHub and click "New Pull Request".
    
    <Warning>
      Make sure your pull request:
      - Has a clear title describing the change
      - Includes a detailed description of what was changed and why
      - References any related issues (e.g., "Fixes #123")
      - Includes screenshots or examples if applicable
      - Passes all existing tests and doesn't break functionality
    </Warning>
    
    **Pull Request Template:**
    
    ```markdown
    ## Description
    Brief description of what this PR does
    
    ## Changes Made
    - Change 1
    - Change 2
    - Change 3
    
    ## Testing
    How did you test these changes?
    
    ## Related Issues
    Fixes #[issue number]
    
    ## Screenshots (if applicable)
    Add screenshots to help explain your changes
    ```
  </Step>
</Steps>

## Code Review Process

After submitting your pull request:

1. **Automated Checks**: Your PR may be checked by automated tools (if configured)
2. **Code Review**: Project maintainers will review your code and may request changes
3. **Discussion**: Be prepared to discuss your implementation and make revisions
4. **Approval**: Once approved, your PR will be merged into the main branch

<Tip>
  Be patient and responsive during the review process. Maintainers may have questions or suggestions for improvement.
</Tip>
